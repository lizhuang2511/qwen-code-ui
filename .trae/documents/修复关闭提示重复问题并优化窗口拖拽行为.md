用户反馈了两个问题：
1.  **保存记录的提示框出现两次**：这很可能是因为我同时将 `on_closing` 绑定到了 `window.events.closing` 和 `api_instance.set_close_handler(on_closing)`。当用户点击原生关闭按钮（虽然被隐藏了，但在某些情况下可能被触发）或通过 `api.quit_app()` 关闭时，可能会触发多次。特别是 `quit_app` 内部已经调用了 callback，如果返回 `True` 继续执行 `destroy()`，可能会再次触发 pywebview 的 closing 事件。
2.  **边界调整和拖拽**：用户希望“边界需要可以调整界面大小，移动窗体只在标题栏区域，不添加系统框”。目前 `frameless=True` 可能会导致无法调整窗口大小（取决于 OS 实现）。在 Windows 上，pywebview 的 `frameless=True` 通常允许调整大小，但如果禁用了 `easy_drag`，可能需要确认行为。用户强调“移动窗体只在标题栏区域”，这已经在前端通过 CSS (`-webkit-app-region: drag`) 实现了，但后端需要配合。

**解决方案计划**：

1.  **解决提示框出现两次的问题**：
    *   **分析**：`window.events.closing += on_closing` 是 pywebview 的原生事件监听。当调用 `window.destroy()` 时，这个事件会被触发。而我们的 `api.quit_app()` 逻辑是：先手动调用 `on_close_callback` (即 `on_closing`)，如果通过，再调用 `window.destroy()`。`window.destroy()` 又会触发 `window.events.closing` (也是 `on_closing`)，导致第二次弹窗。
    *   **修改**：
        *   从 `window.events.closing` 中**移除** `on_closing` 的绑定。
        *   或者，修改 `on_closing` 逻辑，增加一个标志位防止重复执行。
        *   更优雅的做法是：只在 `api.quit_app` 中处理确认逻辑。对于原生关闭（如 Alt+F4），保留 `window.events.closing` 绑定，但要区分来源。
        *   **最佳实践**：由于我们隐藏了系统标题栏，用户主要通过前端按钮关闭，走的是 `api.quit_app`。对于 Alt+F4 等系统级关闭，仍然会触发 `window.events.closing`。
        *   **策略**：修改 `on_closing`，使其不直接绑定到 `window.events.closing`，或者在 `quit_app` 中调用 `destroy` 之前先解绑 `closing` 事件，防止二次触发。
        *   **简化方案**：只保留 `api_instance.set_close_handler(on_closing)`，并在 `main.py` / `start.py` 中**删除** `window.events.closing += on_closing`。
        *   **风险**：如果用户用 Alt+F4 关闭，不绑定 `window.events.closing` 就会直接关闭不提示。
        *   **修正方案**：保留 `window.events.closing += on_closing`。修改 `crates/backend/api.py` 中的 `quit_app`，**不再手动调用回调**，而是直接调用 `window.destroy()`。因为 `window.destroy()` 会自动触发 `closing` 事件，从而调用 `on_closing`，这样就只会出现一次提示。

2.  **解决边界调整问题**：
    *   在 Windows 上，`frameless=True` 的窗口通常是可以调整大小的 (resizeable)。确保 `easy_drag=False`。
    *   用户提到“移动窗体只在标题栏区域”，我们在前端 `CustomTitleBar.tsx` 已经加了 `appRegion: "drag"`。
    *   **确认**：`main.py` 和 `start.py` 中必须设置 `frameless=True` 和 `easy_drag=False`。我之前的修改已经设置了 `easy_drag=False`，但用户提供的 `start.py` 片段显示 `easy_drag=True`（可能是我之前的修改没覆盖到或者用户看的是旧状态？不，我检查了之前的工具调用，应该是改了的。待会再确认一次文件内容）。

**修正后的计划步骤**：

1.  **修改 `crates/backend/api.py`**:
    *   修改 `quit_app` 方法：**删除**手动调用 `self.on_close_callback()` 的逻辑。直接调用 `w.destroy()`。
    *   保留 `set_close_handler` 方法，但它可能不再被 `quit_app` 使用，或者仅用于其他用途。实际上，如果 `quit_app` 只是 destroy，那么 `set_close_handler` 就不需要在 `api.py` 里存 callback 了，因为逻辑都走 pywebview 的事件系统。
    *   **但是**，pywebview 的 `closing` 事件是在 Python 端触发的。当前端调用 `quit_app` -> Python `quit_app` -> `window.destroy()` -> 触发 `closing` 事件 -> 执行 `on_closing` -> 弹出对话框。这个流程是通的，且只会触发一次。
