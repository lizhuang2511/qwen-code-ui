## 目标
- 移除所有“模拟对话/模拟会话”逻辑，只在真实进程成功启动后才显示输入框。
- 在关键流程阶段打印详细的进度与错误信息（事件+日志），方便定位问题。

## 后端改动（无 try/except）
- 移除 `session.py` 中两个模拟分支：
  - 无效工作目录分支：`crates/session.py:148-166`
  - CLI 不存在分支：`crates/session.py:272-297`
- 统一真实流程与失败处理：
  - 会话初始状态：`alive=False`，`ready=False`，`proc=None`。
  - 阶段事件：
    - `starting`（含 `working_directory`）
    - `validating_cli`（含解析到的可执行路径）
    - `spawning_process`（含 `cwd`）
    - `authenticating`（Qwen 握手，含 `base_url`/`model`）
    - `creating_session` → `ready`
  - 失败路径（逐一发出详细事件与日志）：
    - `invalid_working_directory`：保持 `alive=False`，发出 `ai-error-<session_id>`（文本包含目录路径与失败原因）与 `session-progress-<session_id>`（`stage=failed`，带 `details`）。
    - `cli_not_found`：保持 `alive=False`，同样发出错误事件与 `stage=failed`。
    - `spawn_failed`：保持 `alive=False`，发出错误事件与 `stage=failed`。
    - `handshake_failed`（Qwen）：进程已存在但未通过认证，发出详细 `stage=failed` 与错误事件；可按需保留 `alive=True` 表示进程在，但 `ready=False` 禁止交互。
  - 每次 `alive` 状态变更后，发出 `process-status-changed`，确保前端拿到最新 `is_alive`。
- 关键日志（print+事件）：
  - 可执行解析：`resolve_executable` 返回值
  - 实际调用参数：`Popen`/`QwenProcess` 的命令、`cwd`、`model`
  - 握手返回码与目标 URL/model
  - 所有失败原因文本（不使用异常捕获，而用条件与返回码判断）

## 验证路径
- 新建讨论（项目详情页 → 新建）：
  - 后端打印与事件：从 `starting → validating_cli → spawning_process → authenticating → creating_session → ready`，失败时进入 `failed` 并发出 `ai-error-<session_id>`。
  - 前端日志：
    - `[PROCESS-WS] Received real-time status update` 包含新会话 `is_alive=true`
    - `[SESSION-PROGRESS]` 按阶段打印完整负载
  - 结果：仅当 `is_alive=true` 时输入框显示；失败时保持隐藏并可在 UI 或控制台查看失败原因。

## 风险与回退
- 风险：去除模拟后，真实 CLI 缺失或目录异常会导致输入框长时间不可用。
- 缓解：详细进度/错误事件直达前端，用户可快速定位配置问题；必要时在 UI 提示“未启动，查看进度/错误”。
- 回退：如需临时演示环境，可在分支中保留模拟逻辑，主线移除。

## 预期效果
- 只有真实会话进程成功启动时，输入框才渲染；失败路径提供细粒度的阶段与错误信息，便于快速定位问题。
