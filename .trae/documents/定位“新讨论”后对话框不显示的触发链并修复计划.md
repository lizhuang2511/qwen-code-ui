- 点击按钮：`handleNewDiscussion` 调用 `startNewConversation` 并跳转主页 `frontend/src/pages/ProjectDetail.tsx:262-274`
- 建立会话与监听：
  - 创建会话、设为激活、预先绑定事件监听器（`cli-io-*`、`ai-output-*`、`ai-thought-*`、`acp-session-update-*`、`acp-permission-request-*`、`ai-turn-finished-*`）`frontend/src/App.tsx:274-363`、`frontend/src/hooks/useConversationEvents.ts:290-983`

- 发起后端启动：通过 PyWebview JS API 调用 `api.start_session(...)` `frontend/src/lib/api.ts:172-247` → `backend/api.py:28-37`
- 后端会话启动：

- 前端渲染：主页 `HomeDashboard` 根据事件把助手消息与工具调用显示出来；确认请求以紧凑块形式呈现（非模态）`frontend/src/pages/HomeDashboard.tsx:136-170`、`frontend/src/components/common/ToolCallDisplay.tsx:360-701`

## 当前卡点定位
- 进度只到 5%：后端从未派发 `session-progress-<id>`，导致按钮旁的进度一直停留在种子值 `5%`，用户感知为“卡住”。根因：`backend/api.py`、`crates/session.py` 未实现会话阶段进度事件。
- “对话框无法显示”的含义：
  - 新讨论并不会弹模态对话框；工具调用的“权限请求”以消息内的紧凑批准块呈现（非模态）。若未看到该块，多为后端未派发 `acp-permission-request-*` 或前端未收到 `cli-io-*` 的 JSON 流。
  - 在“真 CLI”路径下，当前仅派发 `cli-io-*` 与 `process-started-*`，不保证会出现权限请求；因此用户可能误认为“对话框未显示”。

## 修复实现计划
- 补充后端进度事件：在 `backend/api.py:start_session` 或 `crates/session.py:start_session` 流程内，按照前端 `SessionProgressPayload` 结构派发阶段：
  - `starting(5%)` → `validating_cli(10-20%)` → `spawning_process(30-40%)` → `initializing(60-80%)` → `ready(100%)`
  - 事件名：`session-progress-<sessionId>`，结构字段：`stage/message/progress_percent/details` `frontend/src/types/session.ts:1-17`
  - 统一用现有 `events.emit(event, payload)` 发送 `crates/events.py:4-9`
- 真 CLI 路径的最小握手：
  - 若后端能解析 CLI 输出中的 JSON（ACP 事件），继续依赖 `cli-io-*`；否则在成功启动后手动派发一次 `acp-session-update(tool_call:init)` 与可选的 `acp-permission-request`，保证前端有可视反馈。
- Qwen 参数传递与握手（与前端保持一致）：
  - `backend_config` 中的 `api_key/base_url/model` 已由前端按 Qwen 填充 `frontend/src/App.tsx:300-308`；后端可用于构造一次最小 HTTP 握手（例如 OpenRouter `/v1/chat/completions`）并在成功后派发 `ready(100%)`。
- 保持代码约束：不使用 `try/except`；采用参数校验与早返回。

## 风险与注意事项
- CLI 存在但不输出 ACP 事件时，前端仅显示流式文本；可用一次保底 `ready` 事件保证 UX。
- 事件名需与前端完全匹配；会话 ID 使用 `sanitizeEventName` 规则时保持一致 `frontend/src/hooks/useConversationEvents.ts:10-13`。
- 避免泄露密钥；不要在日志/事件中输出 `api_key`。
