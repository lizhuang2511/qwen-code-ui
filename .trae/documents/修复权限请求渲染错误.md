# 修复权限请求渲染错误的计划

## 问题分析
用户遇到的渲染错误是指 `session/request_permission`（权限请求）消息直接以原始 JSON 格式显示在聊天界面中，而不是渲染成用户友好的“批准/拒绝”卡片。

经过代码分析，主要原因如下：
1.  **前端事件处理遗漏**：在 `frontend/src/hooks/useConversationEvents.ts` 中，`cli-io` 事件监听器没有捕获 `session/request_permission` 类型的消息。这导致 `window.pendingToolCallInput` 未被正确设置，`ToolCallDisplay` 无法获取必要的上下文信息。
2.  **防御性回退机制误判**：前端的“防御性回退”（Defensive Fallback）机制可能错误地将这个 JSON 消息当作了普通的文本回复提取出来，导致它被作为文本消息直接渲染，从而出现了截图中的原始 JSON 块。

## 实施步骤

### 1. 修改 `frontend/src/hooks/useConversationEvents.ts`
我们将更新 `useConversationEvents.ts` 文件中的 `cli-io` 事件监听逻辑：

1.  **捕获权限请求数据**：
    在检查 `jsonData.method` 时，增加对 `session/request_permission` 的支持。
    ```typescript
    if (
      jsonData.method === "requestToolCallConfirmation" ||
      jsonData.method === "session/request_permission"
    ) {
      window.pendingToolCallInput = event.payload.data;
    }
    ```

2.  **防止误判为文本**：
    在提取文本的逻辑中（Fallback 机制），确保 `session/request_permission` 不会被处理。虽然目前的逻辑是白名单机制（只处理 `session/update` 和 `streamAssistantMessageChunk`），但为了稳健性，我们将仔细检查这部分逻辑，确保不会有意外的文本提取。

## 预期结果
1.  聊天界面中将不再直接显示 `session/request_permission` 的原始 JSON 文本。
2.  权限请求将被正确识别并传递给 `ToolCallDisplay` 组件，渲染出包含“允许”和“拒绝”按钮的标准批准卡片。
